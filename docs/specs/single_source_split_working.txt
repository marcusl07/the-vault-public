Single-Source Split Spec
========================

Purpose
-------
The wiki must support splitting one raw source note into multiple atomic wiki pages when that source contains multiple independently reusable ideas.

This is a general wiki rule, not a lecture-note exception:

- splitting is based on atomicity of ideas, not number of source notes
- one raw source may support multiple wiki pages
- topic pages may exist with 2 strong children when the parent label is a clean, non-generic compression

Problem
-------
The current split behavior is constrained by source-count assumptions. That blocks a common maintenance case: one long raw note that clearly contains several atomic ideas cannot be represented as several atomic wiki pages unless multiple source files already exist.

That produces the wrong result for notes such as:

- a lecture note that meaningfully covers multiple standalone ideas
- a long capture that discusses several entities or concepts
- a source that should contribute durable evidence to several different atomic pages

Core Rule
---------
The unit of synthesis is the reusable idea, not the source file.

If one raw note contains several ideas that should stand as separate wiki pages, the system should split that note into multiple atomic pages even when there is only one source file. The same raw source may then appear in the `## Sources` section of multiple child pages.

Definitions
-----------
For this spec:

- an atomic page is one idea that is specific enough to stand on its own and broad enough to remain useful as more sources are added
- a split means one raw source contributes directly to two or more atomic wiki pages
- a child page is an atomic page created or updated from a split
- a reusable idea is an idea that can be linked to directly, found directly in search, and enriched by future sources without collapsing back into a broader parent note
- a strong child is a child page that meets the reusable-idea standard and is not merely a subsection label, storage bucket, or source-shaped wrapper
- a clean compression label is a specific, non-generic parent label that accurately groups the child pages without mirroring source structure and without being nearly identical in meaning to one child

Stability Goals
---------------
Single-source splitting should improve the structure of the wiki, not just increase page count.

The system should preserve these invariants:

- prefer fewer stronger pages over many thin pages
- do not create sibling pages that are mostly the same idea with minor rewording
- do not split when the child set is less stable than the original atomic page
- repeated runs over unchanged source material should normally preserve the same split outcome
- query-time and maintenance-time behavior should follow the same split rules

Split Decision Rules
--------------------
Split a raw note when all of the following are true:

- the note contains two or more independently reusable ideas
- each candidate child would be useful as its own atomic page
- each candidate child can be grounded directly in the raw source
- the candidate children are meaningfully distinct from one another
- the split would produce a more stable structure than keeping the note atomic

Do not require any of the following:

- multiple raw source files
- explicit section headings
- one isolated source subsection per child page

Do not split when any of the following are true:

- the note is still one coherent atomic page despite having several subpoints
- the proposed children are generic buckets, storage labels, or source-shaped wrappers
- the proposed children are not grounded in the source
- the proposed children are thin, heavily overlapping, or differ mainly by wording
- the distinction between sibling pages cannot be explained cleanly in one sentence per sibling
- the split would create more fragmentation than retrieval value

Operational Test
----------------
When deciding whether to split, evaluate each candidate child page with this test:

1. Would this idea be worth linking to directly from another wiki page?
2. Would this idea be worth finding on its own in search?
3. Would this idea still make sense as a stable page if the source note later gained more detail?

Evaluate these questions against retrieval value beyond the current source note, not just local section structure inside that note.

Each proposed child page must pass questions 1 and 3.

Question 2 is a useful corroborating signal, not a substitute for the other two.

If fewer than 2 proposed child pages pass that threshold, do not split.

If one candidate passes and another fails, do not split into that pair as proposed. Either keep the note atomic or revise the candidate set to include only child pages that independently satisfy the split rules.

Minimum substance rule:

- Do not create a child page when the source supports only a passing mention or a single incidental sentence about that idea.
- Each child page must be supported by enough source material to justify at least one meaningful note or fact on that page.

Anti-fragmentation rule:

- Prefer the smallest child set that captures the durable distinctions in the source.
- Do not split a source into many children when a smaller number of stronger children would preserve the same retrieval value.

Overlap rule:

- Child pages may share the same source.
- Child pages should not substantially duplicate each other's notes.
- If two proposed children overlap so heavily that their distinction is unstable, merge them or keep the parent atomic.

Examples
--------

- Split: one note about fast-food history that contains durable material on McDonald's history and Burger King history
- Split: one note that substantively covers both iterators and generators as separate reusable concepts
- Do not split: one note whose subpoints are all still part of one atomic note about a single concept

Lecture Notes
-------------
Lecture notes are not special-cased.

- Apply the same atomicity rules to lecture notes and non-lecture notes.
- A lecture note may remain atomic if it is still one coherent reusable note.
- A lecture note may split if it clearly contains multiple independently reusable ideas.

The source being a lecture is not, by itself, a reason to preserve or force a split.

Source Reuse And Provenance
---------------------------
One raw source may support multiple atomic pages.

- After a split, the same raw note may be listed in the `## Sources` section of multiple child pages.
- Each child page should list the raw source directly when that source grounds the child page.
- Do not force one raw source to have exactly one owning wiki page.
- Do not hide provenance only on the parent topic page.
- Do not list the same raw source on a child page when that source contributes only incidental mention-level support.

If the raw source includes an external URL and a child page is grounded in that source, the child page should include the literal URL in `## Sources` following the normal wiki source-format rules.

Child Page Content
------------------
Child pages may contain grounded synthesis.

- A child page does not need to be a direct passage extract.
- The system may write compact synthesized notes for each child page as long as the content is grounded in the shared raw source.
- Synthesis should remain faithful to the source and should not invent unsupported claims.

Parent And Topic Behavior
-------------------------
When a split produces multiple atomic children, the original umbrella page should become a topic page only when the parent label is still a clean compression of the children.

Topic-page rule:

- Topic pages may exist with 2 or more strong child pages.
- The parent label must be specific, clean, and non-generic.
- Do not create or preserve topic pages with generic labels such as `notes`, `misc`, `week-3`, `lecture-5`, or similar source-shaped buckets.
- Do not create a topic page when the parent label is nearly synonymous with one child page.

If the original parent label is generic or source-shaped, do not preserve it as a topic page just because a split occurred.

When a split occurs, the system should resolve the original page in this priority order:

- If the original page title is still a valid atomic page, keep it as an atomic page and update it normally.
- If the original page title is a clean compression of 2 or more strong children, convert it to a topic page.
- If the original page title is generic or source-shaped and is not still a valid atomic page, deprecate that page rather than preserving it as a topic page.

If a title could plausibly qualify as both a valid atomic page and a clean compression label, prefer keeping it atomic.

Deprecation here means:

- do not delete the page automatically
- add a brief note at the top that the page has been superseded by more atomic children
- link directly to the replacement child pages

Recommended deprecation note format:

- `Deprecated: superseded by [[child-a]] and [[child-b]].`

Implication:

- the general wiki rule is no longer "topic pages require at least 3 children"
- the general wiki rule becomes "topic pages require at least 2 strong children plus a clean compression label"

Update Behavior
---------------
Single-source splitting should be available wherever the wiki is actively maintained.

- It should be available during ingest of new notes.
- It should also be available in maintenance paths that revisit existing wiki pages.
- It should not be treated as bootstrap-only behavior.

Idempotency And Re-evaluation
-----------------------------
The split decision should be stable under repeated runs.

- Re-running ingest or maintenance on unchanged source material must preserve the same page structure unless the previous structure violates the current split rules.
- Do not re-split an unchanged source into a different child set unless the previous structure is clearly deficient under the current split rules.
- New sources may justify later refinement, but unchanged evidence alone should not cause routine oscillation between atomic and split forms.
- If the best boundary between children remains unclear after applying the split rules, prefer the more stable structure, which is usually to keep the note atomic or use fewer children.

Query-Time Behavior
-------------------
Queries may produce durable wiki updates, but only narrowly.

- A query may trigger a split or related structural fix only when there is a clear structural deficiency in the existing wiki and the change is strongly grounded in existing sources.
- Query-time writes should be narrow and auditable, not broad opportunistic maintenance.
- Query-time writes must not modify `raw/`.
- Query-time writes should remain logged as `query` entries with a short mutation note.
- Query-time writes should affect only the pages directly involved in the split and its immediate parent/child structure.

For this spec, a related structural fix means split-adjacent work only, such as:

- converting an umbrella page into a topic page
- deprecating a superseded generic umbrella page
- updating links among the affected parent and child pages

It does not include broad discovery of unrelated new pages during query handling.

If query-time confidence is not high enough to choose stable child boundaries, do not mutate structure during the query.

This allows the wiki to improve through use without turning query execution into unrestricted background maintenance.

Rollout
-------
The intended steady-state behavior is default-on.

- Validate behavior in a sandbox wiki first.
- Before default-on rollout, confirm in the sandbox wiki that the behavior satisfies every acceptance criterion in this spec.
- Before default-on rollout, confirm that test cases include at least:
  - one single raw note that should remain atomic
  - one single raw note that should split into multiple atomic children
  - one case where a 2-child topic page is valid
  - one case where a generic umbrella label is rejected or deprecated
- If those checks pass, ship single-source splitting as the normal behavior rather than a permanent opt-in feature.

Implementation Tasks
--------------------
Implement this spec as a sequence of atomic tasks rather than one large change.

Recommended task order:

1. split decision
   - implement the decision path that determines whether one raw source should remain atomic or split
   - expose enough intermediate reasoning or debug output to inspect candidate children, grounding, and rejection reasons during sandbox validation

2. split realization during ingest or maintenance
   - implement creation or update of child pages after a split decision
   - implement parent-page resolution: keep atomic, convert to topic, or deprecate
   - keep this step limited to ingest and maintenance paths first

3. provenance reuse
   - implement reuse of one raw source across multiple child pages
   - enforce the rule that incidental mention-level support does not justify adding the source to a child page

4. stability checks
   - verify that rerunning the same unchanged source is a no-op structurally
   - verify that near-duplicate or heavily overlapping child sets are rejected or merged according to this spec

5. query-time structural writes
   - implement split-adjacent query-time writes only after the ingest and maintenance behavior is stable
   - keep query-time writes narrow, logged, and limited to directly affected pages

Acceptance Criteria
-------------------
The spec is satisfied if all of the following are true:

- one raw note can yield multiple atomic wiki pages
- split decisions depend on reusable ideas rather than source-count
- lecture notes are eligible for the same split behavior as any other source
- the same raw source can appear on multiple child pages after a split
- child pages can contain grounded synthesized notes
- a 2-child topic page is allowed when the parent label is a strong, non-generic compression
- generic parent labels are still rejected
- query-time structural writes, when they occur, are narrow, grounded, and logged
- a coherent atomic note is not split solely because it is long or has headings
- sibling child pages are meaningfully distinct rather than near-duplicates
- incidental mention-level grounding does not by itself create a child page or source entry
- unchanged source material does not normally cause repeated restructuring into different child sets

Non-Goals
---------
- forcing every long note to split
- requiring heading-based extraction only
- preserving source-shaped umbrella pages when they are not real atomic notes
- turning query-time behavior into unrestricted whole-wiki maintenance
- defining full cross-source conflict resolution or whole-wiki de-duplication policy
